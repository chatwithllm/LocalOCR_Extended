## Critical context (architectural mismatch flagged at Gate 1)

The runbook describes the codebase as "Flask + Jinja2 (~57% HTML, ~42% Python)" with "templates" under `src/templates/` to convert one workspace at a time. **That's not what's here.**

What actually exists:

- A **single-file Vanilla-JS SPA** at `src/frontend/index.html` (~38 K lines, all 13 workspaces in one file).
  - Workspaces are `<div class="page" id="page-X">` blocks shown/hidden via the `.active` class — Flask just serves this static HTML and a JSON API.
  - There is no `src/templates/`. There are no Jinja2 partials. The `init` skill, `docs/ARCHITECTURE.md`, and the `README.md` (line 7: "Vanilla-JS SPA") all confirm this.
- Existing styling pipeline that already supports opt-in themes:
  - `design/design-tokens.json` — JSON token source with a `themes.<name>` block per opt-in theme.
  - `scripts/build_tokens.py` — compiles JSON → `src/frontend/styles/tokens.generated.css`. Each `themes.<name>` entry becomes a `:root[data-theme="<name>"] { ... }` override block.
  - `src/frontend/styles/design-system.css` — handwritten component-scoped CSS. Includes Clay-themed variants like `:root[data-theme="clay"] .btn { ... }`.
  - Inline `<style>` in `index.html` (top of the file) — bootstrap copy of the same tokens, used during pre-paint to avoid FOUC. Mirror of tokens.generated.css.
- Existing precedent: the **Clay theme** was added in this exact pattern (commits `a492963`, `bf6caf7`, `539cc4e`, `3d33575`). Apple is the base, Clay is opt-in. Theme picker in Settings (line 15007) + cycling toggle in the sidebar.

So the runbook's "convert per template" framing must be **reinterpreted** as:

| Runbook concept | Actual artifact in this repo |
|---|---|
| `src/templates/layouts/authenticated.html` | The base CSS in `index.html` `<head>` + linked `design-system.css` + `tokens.generated.css` |
| Workspace template | The `<div class="page" id="page-X">` block inside `index.html` plus its handlers |
| `partials/badge.html` | A reusable CSS class (e.g. `.badge--domain.badge--grocery`) wrapping an inline `<span>` |
| Workspace conversion | Verifying the existing markup uses the right component classes; adding new classes where patterns are missing |

**No DARK_REDESIGN.md** exists in the repo (Hard Rule 6 confirmed moot via `find . -name "DARK_REDESIGN*"`). The Notion spec at `design/Design System inspired by Notion.md` references it as a sibling integration prompt, but this runbook itself plays that role.

## Approach

**Add Notion as an opt-in theme** following the exact Clay precedent, then layer in the workspace-specific patterns from §11 of the spec that don't yet exist anywhere (domain badges, OCR-reviewed pill, receipt editor structural classes, guest banner, Kitchen Display layout class). The patterns get added as base chrome (so all themes inherit them) but tuned via theme-scoped overrides for Notion specifically.

This means:
- Apple (default) and Clay continue to work unchanged.
- Notion becomes the fourth/fifth picker option (`notion`, `notion-dark`).
- The new component patterns from §11 land as base classes; non-Notion themes get sensible default styling, Notion gets the spec-faithful styling.
- Operational plumbing (backend, migrations, MQTT, OCR, auth, Docker) is not touched.

### CSS strategy decision

**Plain CSS + custom properties via the existing two-layer system** (`tokens.generated.css` + handwritten `design-system.css`).

Justification: This is the established pattern in the repo — Apple, Apple-dark, Clay, and Clay-dark all use it. Introducing a new build tool (Tailwind, PostCSS plugins, CSS-in-JS) for one more theme would be needless churn. The token pipeline already handles theme-scoped overrides cleanly via `:root[data-theme="<name>"]`. Sticks with what already ships.

### Token file location

- **Token source:** `design/design-tokens.json` — add a `themes.notion` and `themes.notion-dark` block alongside the existing `themes.clay` and `themes.clay-dark`.
- **Generated CSS:** `src/frontend/styles/tokens.generated.css` — regenerated by `python3 scripts/build_tokens.py`. **Never hand-edited.**
- **Inline mirror:** `src/frontend/index.html` `<style>` block at the top — mirror the new theme's overrides inline so pre-paint switching works. (Same pattern Clay used — see commit `a492963`.)
- **Component CSS:** `src/frontend/styles/design-system.css` — append a Notion-themed section with `:root[data-theme="notion"] .btn { ... }` style scoped overrides.

### Workspace conversion order

Following the runbook's suggested order, mapped to actual `#page-X` blocks:

1. **Tokens & base layout** — `themes.notion` JSON block, regenerate `tokens.generated.css`, mirror inline, add picker entry, add to theme cycle. No visible change yet for non-Notion users.
2. **Sidebar / nav chrome** — Notion-scoped sidebar styling: `15px / 500`, hover/active states, focus rings.
3. **`#page-dashboard`** — stat cards, household ranking card with long-press reveal, budget cards using §11 spec.
4. **`#page-upload`** — intent picker pills (Auto/Grocery/Restaurant/General Expense), drop-zone styling, OCR step labels.
5. **Receipts list (`#page-receipts`)** — whisper dividers, domain badges per row, filters.
6. **Receipt correction editor** — two-column structural rebuild with sticky footer, image frame, OCR-reviewed pill.
7. **`#page-inventory`** (Inventory + Products toggle) — grouped rows, search magnifier mobile pattern, no-result manual-add.
8. **`#page-shopping`** — `Open / Estimate / Close` summary strip already exists; verify Notion polish + QR helper.
9. **`#page-budget`** — target-in-label, 40px value, progress bar with state colors.
10. **`#page-analytics`** — 3-color max charts, gridlines, legend below on mobile.
11. **`#page-restaurant`** — domain-appropriate badges; no inventory side effects.
12. **`#page-expenses`** — same chrome, expense domain badges.
13. **`#page-contributions`** — pink streak/rank badges sparingly per spec.
14. **`#page-settings`** — trusted device cards with scope badges, Theme picker entry already exists, add Notion options.
15. **Kitchen Display variant** — `body.is-kitchen-display` class + 1.25× type + reduced motion.
16. **Guest mode polish** — banner + sign-in-instead-of-submit gating (see Open Questions).

Each step ends with one or more conventional commits (`feat(tokens):`, `feat(layout):`, `feat(dashboard):`, etc.) and a push.

## Known risks and open questions

1. **Architecture mismatch (flagged above)** — runbook expects Jinja2 templates; reality is single-file SPA. Plan reframes the work as theme + scoped CSS additions following the Clay precedent. **Confirm this reframing is acceptable before I start.**

2. **Notion as opt-in theme vs as new default** — I'm planning opt-in (sit alongside Apple/Clay in the picker). This matches the Clay precedent and is lower-risk. If you want Notion to *replace* Apple as the unconditional default, say so at Gate 1 and I'll adjust.

3. **Guest demo mode is minimal today** — I found one mention of "sample data" copy in `index.html` (line 12647) but no read-only enforcement, no banner, no sign-in modal redirection. PRD §5A requires it. The runbook expects guest mode to be polishable — but if it's not actually implemented, "polish" means *building* it, which leans toward backend/auth changes. **I will style the banner + ghost-out CTAs in markup, but if backend gating is needed, I'll defer per scope-creep protocol.**

4. **Kitchen Display layout depth** — `isKitchenDisplayTrustedDevice()` and a few class hooks exist, but I haven't seen a true "1.25× type, ≥56px touch zones" layout variant. Plan adds `body.is-kitchen-display` styling; this is purely CSS-side.

5. **Restaurant workspace is hidden by default** — `nav-restaurant` has `style="display:none"`. Not sure if this is dormant work or intentionally feature-flagged. **Plan assumes the workspace exists and will style it; visibility toggling is unchanged.**

6. **`design/Design System inspired by Notion.md` is currently untracked** in the repo. I'll add it to the redesign branch's first commit (the plan commit). Confirms it as the authoritative spec going forward.

7. **No automated visual regression tests** — quality gate per workspace relies on careful CSS reading + spec cross-check. Smoke test at Gate 2 is the human verification.
